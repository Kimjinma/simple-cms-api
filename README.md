# [백엔드 과제 제출] (주)맑은기술 지원자 김진용

안녕하세요. 2026년도 백엔드 개발자 과제를 제출합니다.
본 프로젝트는 요구사항의 정확한 구현뿐만 아니라, 향후 유지보수와 확장이 용이한 API 설계 및 안전한 권한 검증 처리에 중점을 두어 개발했습니다.

---

## 1. 기술 스택
- **Language & Framework:** Java 25, Spring Boot 4
- **Database:** H2 Database (In-Memory), Spring Data JPA
- **Security:** Spring Security, JWT (JSON Web Token)
- **Build & Tools:** Gradle, Lombok

---

## 2. 실행 가이드 및 테스트 환경, REST API 문서

REST API 문서로는 프로젝트 최상위 폴더 CMS_API_Postman_Test.json 파일을 확인해주시면 감사하겠습니다
평가자님의 원활한 테스트를 위해 초기 더미 데이터를 세팅해 두었습니다. 
서버 구동 시 `data.sql`을 통해 아래의 테스트 계정과 15개의 페이징 테스트용 게시글이 자동 생성됩니다.


| 권한 | Username | Password | 비고 |
|---|---|---|---|
| **관리자 (ADMIN)** | `admin` | `1234` | 모든 게시글 수정/삭제 가능 |
| **일반 유저 (USER)** | `user1` | `1234` | 본인 작성 글만 수정/삭제 가능 |

* **토큰 세팅 방법**  <br>
    - 1. `/api/auth/login` 에 위 계정 정보로 POST 요청을 보내어 `token` 값을 획득합니다. <br>
    - 2. 게시글 작성/수정/삭제 요청 시 헤더에 `Authorization: Bearer {발급받은토큰}` 형태로 삽입하여 테스트하실 수 있습니다.

## 3. 핵심 아키텍처 및 설계 의도 

### A. 상태 비저장(Stateless) 기반의 JWT 인증 도입
* **설계 의도:** 초기 세션(Session) 방식 대신, 향후 다중 서버 환경이나 프론트엔드/백엔드 분리 아키텍처로의 확장성을 고려하여 JWT 기반의 인증 방식을 채택했습니다.
* **구현:** `JwtProvider`를 통해 토큰을 발급/검증하며, 커스텀 필터(`JwtAuthenticationFilter`)를 구현하여 토큰 검증 후 `SecurityContextHolder`에 인증 객체를 할당하도록 구성했습니다.

### B. 계층 간 데이터 보호를 위한 DTO(Data Transfer Object) 분리
* **설계 의도:** 엔티티(Entity)의 무분별한 노출을 막고 API 스펙의 변경을 방어하기 위해 DTO를 철저히 분리했습니다.
* **구현:** 요청의 목적(생성, 수정, 로그인 등)에 따라 Request DTO를 개별 정의하여, 비즈니스 로직에 필요한 데이터만 안전하게 전달받고 검증(Validation)할 수 있도록 설계했습니다.

### C. 비즈니스 계층(Service)에서의 안전한 권한 검증 처리
* **설계 의도:** 컨트롤러 단이 아닌, 실제 데이터를 조작하는 서비스 계층에서 인가(Authorization) 처리를 수행하여 보안을 강화했습니다.
* **구현:** 게시글 수정/삭제 로직 수행 시, 요청자의 `username`과 `isAdmin` (관리자 여부) 값을 주입받아 **"작성자 본인이거나 관리자인 경우에만"** DB 조작이 가능하도록 예외 처리 로직을 구현했습니다.

## 4. 참고 자료 및 도구 활용
본 과제는 개발 생산성 향상과 설계 타당성 검증을 위해 AI 코딩 어시스턴트(Gemini)를 적극적으로 활용했습니다. 단, 단순한 코드 복사/붙여넣기를 지양하고 다음과 같이 활용하며 철저하게 코드 주도권을 확보했습니다.

* **효율적인 초기 세팅 및 보일러플레이트(Boilerplate) 생성:**
    * 반복적인 Entity, DTO, Repository 클래스 생성 등 초기 뼈대 구축 시간을 단축하여, 핵심 비즈니스 로직(권한 검증, 예외 처리) 고민에 더 많은 시간을 투자했습니다.
* **아키텍처 설계의 트레이드오프(Trade-off) 검증:**
    * 단순 구현을 넘어, 상태 비저장(Stateless) 아키텍처를 위한 Session vs JWT 방식의 장단점, 단방향 암호화(BCrypt)의 필요성, API 스펙 보호를 위한 DTO 분리 목적 등 백엔드 핵심 설계 원칙에 대해 AI와 기술적인 검토를 진행했습니다.
* **엄격한 코드 리뷰 및 내재화 (Code Ownership):**
    * AI가 제안한 `JwtAuthenticationFilter` 및 글로벌 예외 핸들링 코드 등을 맹목적으로 수용하지 않았습니다. 
    * 프로젝트의 요구사항(H2 환경, 역할 기반 인가 처리)에 맞게 불필요한 잉여 코드를 걷어내고 최적화하는 리팩토링 과정을 거쳤으며, 데이터의 전체 생명주기(Request -> Controller -> Service -> DB -> Response)를 완벽하게 통제하고 디버깅할 수 있도록 구현 했습니다.
